一句话判断：

> 当前目录的“训练和验证工具链”是有价值的，但“可上板的真实模型资产”还没有完成。MP157 上应先跑轻量分类模型，分割模型只做辅助可视化或复判。

## 3. 分类模型如何检测出好坏

这里的“检测好坏”不是目标检测框，也不是像素级分割，而是**整图二分类**。

### 3.1 训练阶段逻辑

分类数据应按下面结构放置：

```text
datasets_classify/
  train/
    good/
    bad/
  val/
    good/
    bad/
```

训练脚本 `train_classify.py` 的核心流程：

| 步骤 | 脚本行为 | 作用 |
|---|---|---|
| 1 | 使用 `torchvision.datasets.ImageFolder` 读取 `train` 和 `val` | 根据文件夹名自动生成类别标签 |
| 2 | 图片统一 `Resize((224,224))` | 匹配 MobileNetV3-Small 输入尺寸 |
| 3 | 做 ImageNet 标准归一化 | 匹配预训练权重分布 |
| 4 | 加载 `mobilenet_v3_small` 预训练模型 | 用轻量 CNN 提取零件表面特征 |
| 5 | 把最后分类层改成 2 类 | 输出 good/bad 两个类别 |
| 6 | 使用 `CrossEntropyLoss` | 让模型学习正确类别 |
| 7 | 自动按类别数量加权 | 缓解 good/bad 样本数量不平衡 |
| 8 | 用验证集准确率保存 `best_model.pth` | 选择验证效果最好的模型 |

分类模型学到的是：

| 输入现象 | 模型学习目标 |
|---|---|
| 良品图片 | 输出 good 类概率更高 |
| 缺陷图片 | 输出 bad 类概率更高 |
| 划痕、污点、裂纹、压伤、毛刺等整体视觉差异 | 通过 CNN 特征提取变成 good/bad 判定 |

### 3.2 推理阶段逻辑

`infer_classify.py` 的当前推理流程：

| 步骤 | 行为 |
|---|---|
| 1 | OpenCV 读取图片或摄像头帧，格式是 BGR |
| 2 | resize 到 `224x224` |
| 3 | BGR 转 RGB |
| 4 | 做 ImageNet 均值方差归一化 |
| 5 | ONNXRuntime 执行模型，得到形状 `(1,2)` 的 logits |
| 6 | 对 logits 做 softmax，得到两个概率 |
| 7 | 用 `argmax` 选择概率最大的类别 |
| 8 | 在画面上显示 GOOD 或 BAD |

可以理解成：

```text
摄像头图像
  -> 224x224 预处理
  -> MobileNetV3-Small
  -> [bad_score, good_score]    ← ImageFolder 按字母排序：bad=0, good=1
  -> softmax
  -> [bad_prob, good_prob]
  -> 概率更大的类别就是最终结果
```

示例：

| bad 概率 | good 概率 | 当前脚本结果 |
|---:|---:|---|
| 0.08 | 0.92 | GOOD |
| 0.65 | 0.35 | BAD |
| 0.49 | 0.51 | GOOD |

### 3.3 类别映射风险（已彻底修复）

~~这是当前分类链路最危险的一点。~~ 已于 2026-05-07 彻底修复。

`train_classify.py` 使用的是：

```python
datasets.ImageFolder(train_dir, transform=train_transform)
```

`ImageFolder` 默认会按文件夹名字母序生成标签。因为 `bad` 字母序排在 `good` 前面，所以会得到：

| 文件夹 | 训练标签索引 |
|---|---:|
| `bad` | 0 |
| `good` | 1 |

`infer_classify.py` 已修正为：

```python
CLASS_NAMES = ["bad", "good"]
BAD_CLASS_INDEX = CLASS_NAMES.index("bad")
GOOD_CLASS_INDEX = CLASS_NAMES.index("good")
```

现在推理时的映射与训练一致：

| 输出索引 | 推理脚本解释 |
|---:|---|
| 0 | bad |
| 1 | good |

当前推理脚本不再在业务逻辑中手写 `pred_class == 1` 判断坏件，也不再直接把 `probs[1]` 当作 bad 概率，而是通过 `BAD_CLASS_INDEX`、`GOOD_CLASS_INDEX`、`is_bad_prediction()`、`get_class_probability()` 和 `result_name()` 统一解释类别。

**修复前的风险**（已解决）：

| 实际模型输出 | 旧脚本显示 | 后果 |
|---|---|---|
| index 0 是 bad | 显示 good | 坏件可能被放行 |
| index 1 是 good | 显示 bad | 良品可能被误剔除 |

**后续可选优化**：

| 优化项 | 说明 |
|---|---|
| 保存类别映射 | `train_classify.py` 保存 `class_to_idx` 到 checkpoint 或 JSON |
| 导出时带出映射 | `export_classify_onnx.py` 在 ONNX 旁边保存 `defect_classifier_labels.json` |
| 推理时读取映射 | `infer_classify.py` 从导出的标签文件读取，而非写死 |
| 阈值逻辑 | 当前 `BAD_THRESHOLD=0.5` 没有真正参与最终判定，建议实现 `bad_prob >= threshold` 判 bad |

已新增回归测试：

| 测试文件 | 覆盖内容 |
|---|---|
| `tests/test_infer_classify.py` | 验证 `bad=0` 高置信时输出 BAD、`good=1` 高置信时输出 GOOD，并确认 bad/good 概率取对索引 |

## 4. 分割模型在 MP157 上是否会卡

结论：**会有明显卡顿风险，不能默认当作每帧实时主链路。**

### 4.1 为什么分割比分类重

| 对比项 | 分类模型 | 分割模型 |
|---|---|---|
| 输出 | 2 个类别概率 | `5 x 224 x 224` 像素级结果 |
| 任务 | 判断整张图好坏 | 判断每个像素属于哪个缺陷类 |
| 后处理 | softmax + argmax 两个数 | 每个像素 argmax、mask resize、颜色叠加、缺陷像素统计 |
| 标注成本 | 只需整图 good/bad 标签 | 需要画缺陷轮廓 mask |
| 运行压力 | 小 | 大 |
| 适合 MP157 主链路 | 适合 | 不建议 |

### 4.2 当前分割模型体积

| 模型 | FP32 ONNX | INT8 后预估 | MP157 建议 |
|---|---:|---:|---|
| MobileNetV2 + UNet | 25.27 MB | 约 6-8 MB | 不建议主链路 |
| MobileNetV3-Small + UNet | 13.70 MB | 约 3-5 MB | 可按需尝试 |
| MobileNetV3-Small 分类 | 预计约 5 MB | 约 1-2 MB | 推荐主链路 |

### 4.3 MP157 上的实际风险

STM32MP157 是双核 Cortex-A7，没有专门的神经网络加速器。分割模型在板端的风险主要是：

| 风险 | 具体表现 |
|---|---|
| CPU 占用高 | 推理时 UI、摄像头采集、通信、文件保存可能被拖慢 |
| 单帧耗时长 | 无法达到 20-30 FPS，只能低帧率显示 |
| 内存压力 | ONNXRuntime、模型权重、中间激活、OpenCV 图像缓冲会同时占内存 |
| 后处理也耗时 | mask resize、overlay、保存 PNG/JPG 都会占 CPU |
| 发热和稳定性 | 长时间高负载可能影响整机稳定性 |

建议判断标准：

| 实测指标 | 建议目标 |
|---|---|
| 分类 INT8 单帧耗时 | 尽量低于 50 ms |
| 分割 INT8 单帧耗时 | 如果超过 150-200 ms，就只适合按需触发 |
| UI 是否卡顿 | Qt 界面不能明显掉帧或假死 |
| 内存占用 | 推理时仍要给 Qt、摄像头、日志、通信留余量 |
| 连续运行 | 至少连续运行 30-60 分钟观察稳定性 |

## 5. 分类和分割能否同时存在

可以同时存在，但建议按“主链路 + 辅助链路”组织。

### 5.1 推荐运行架构

```text
摄像头采集
  -> OpenCV ROI/居中裁剪
  -> 分类模型 MobileNetV3-Small INT8
      -> GOOD：快速放行，只保存必要日志
      -> BAD 或低置信度：触发分割模型
          -> 输出 mask
          -> 生成 overlay 图
          -> 保存缺陷证据图
          -> Qt 界面显示缺陷区域
```

### 5.2 三种共存模式

| 模式 | 运行方式 | 优点 | 缺点 | 推荐程度 |
|---|---|---|---|---|
| 分类常驻，分割按需触发 | 每帧跑分类，只有 bad/低置信度才跑分割 | 速度和效果平衡最好 | 需要调度逻辑 | 推荐 |
| 用户手动打开分割 | 平时只分类，点击查看时跑分割 | 最省资源 | 不能自动记录所有缺陷区域 | 推荐 |
| 分类和分割每帧都跑 | 每帧同时执行两个模型 | 结果最完整 | MP157 很可能卡 | 不推荐 |

### 5.3 板端部署建议

| 资源 | 建议 |
|---|---|
| 模型加载 | 分类模型常驻内存，分割模型可按需加载或后台低频运行 |
| 线程调度 | 摄像头采集、推理、Qt 显示分线程，用队列传递最新帧 |
| 推理频率 | 分类可尽量高频，分割限制频率，例如 1-3 FPS 或 bad 时触发 |
| 图像保存 | 保存前确认文件写完，必要时 `fsync` 或 `sync`，不要用固定 sleep 冒充保存完成 |
| 阈值策略 | 分类 bad_prob 设置安全阈值，低置信度进入人工复核或分割复判 |

## 8. 建议的最终系统策略

推荐把整个检测系统分成两层：

| 层级 | 模型/算法 | 作用 |
|---|---|---|
| 第一层，实时主链路 | OpenCV ROI + MobileNetV3-Small INT8 分类 | 快速判断 good/bad，控制分拣节拍 |
| 第二层，证据增强链路 | MobileNetV3-Small UNet INT8 分割 | bad 或低置信度时输出缺陷区域图 |
| 辅助规则 | 面积阈值、轮廓规则、保存证据图 | 降低误判，提高可解释性 |

最终效果应该是：

| 场景 | 动作 |
|---|---|
| 良品且置信度高 | 分类模型快速判 GOOD，直接放行 |
| 坏品且置信度高 | 分类模型判 BAD，触发分拣，必要时保存图片 |
| 置信度接近阈值 | 触发分割或人工复核 |
| 需要查看缺陷位置 | 运行分割模型，生成 mask 和 overlay |
